# 📌 1. 왜 Toss Payments를 선택했는가?

결제 PG(Payment Gateway)는 실제 운영 환경뿐 아니라 테스트·개발 환경에서도 매우 중요한 인프라 요소입니다.  
이번 프로젝트에서 Toss Payments를 선택한 이유는 다음과 같습니다.

---

## 🚀 1.1 통합 결제 API의 단순성과 개발 효율성

Toss Payments는 다른 PG사보다 **단일 API 구조가 직관적**이며 다음과 같은 장점이 있음:

- Restful 기반 API 구조가 깔끔함
- 요청/응답 구조가 명확하여 개발 속도 증가
- 공식 문서 품질이 매우 뛰어남  
- 에러 코드 체계가 정리되어 있어 디버깅이 쉬움

**→ 개발자가 빠르게 구현하고 유지보수까지 고려하기 좋은 구조**

---

## 👨‍💻 1.2 개발 모드(테스트 모드)가 매우 잘 되어 있음

다른 PG들은 종종 테스트 환경과 운영 환경의 설정이 크게 달라 배포 전까지 정상 동작을 확신하기 어려움.  

반면 Toss는:

- 테스트 Key 로 실제 결제와 거의 동일한 환경 제공
- 테스트 카드 번호 제공
- Callback(success/fail URL) 구조가 운영과 동일
- Webhook 기능 테스트 가능

**→ 실제 운영과 거의 동일한 흐름을 사전에 확인 가능**

---

## 💸 1.3 무료이며 계약 없이 바로 사용 가능 (테스트 환경)

많은 PG사는 테스트 환경 사용 시에도  
계약/심사/사업자 등록이 필요한 경우가 많음.

Toss Payments는:

- 사업자 등록 없이 개발자 계정만 있으면 바로 테스트 가능  
- Webhook도 무료로 테스트 가능  
- 카드/계좌/간편결제 등 거의 모든 기능 테스트 가능

**→ 프리프로덕션 개발에 최적**

---

## 🌐 1.4 한국 내 사용자 경험 및 모바일 환경 최적화

실제 사용자 관점에서도 Toss는 다음 이유로 가장 많이 선택됨:

- 모바일 UI/UX가 가장 우수  
- 결제 과정이 빠르고 자연스러움  
- 사용자 신뢰도가 높으며 이탈율 감소  
- Apple Pay, Naver Pay, Kakao Pay 등 다양한 간편결제를 함께 통합 제공

**→ 결제 성공률이 높은 PG사**

---

## 🔒 1.5 보안 처리 및 Webhook 연동 용이

Toss Payments는 다음 기능을 기본적으로 제공:

- 시그니처 기반 Webhook 검증  
- Redirect + Server confirm 2단계 보안 구조  
- 모든 요청을 HTTPS 기반으로 처리  
- 민감 정보 직접 저장 금지 → PCI-DSS 대응 불필요

**→ 서비스가 민감 정보를 직접 다루지 않아 보안 부담 감소**

# 📊 2. 한국에서 사용 가능한 PG사 비교

| PG사 | 장점 | 단점 |
|------|------|------|
| **Toss Payments** | 개발 편의성 최고, 문서 품질 우수, 테스트 환경 강력, 간편결제 강점 | 브랜드 인지도는 높으나 일부 기업은 전통 PG 선호 |
| **KG이니시스** | 전통적으로 가장 많이 사용, 기능 다양 | API 복잡, 테스트 환경 불안정 |
| **KCP** | 다양한 기능 제공 | 문서 난해, 구현 복잡 |
| **나이스페이** | 안정성 높음 | 초기 설정이 번거롭고 문서 가독성이 낮음 |
| **토스(간편결제만)** | UX 최강 | PG 전체 기능을 제공하지 않음 |

**→ 개발 관점에서는 Toss Payments가 가장 빠르게 구현 가능**

---

# 🧩 3. 이번 프로젝트에 Toss Payments를 선택한 직접적인 이유

### ✔ 실 운영이 아닌 테스트 목적이 가장 중요  
### ✔ 빠른 개발 속도와 유지보수 효율성 필요  
### ✔ Gateway + Order + Payment MSA 구조에 적합  
### ✔ Webhook 기반 이벤트 처리 시나리오가 명확  

특히 이번 프로젝트는 MSA 구조이며 Order-Service와 Payment-Service 간의 협력이 필요한 상황.

Toss Payments는 서버 confirm 로직이 단순하여 MSA에서 다음 장점이 있음:

- Order → Payment → Toss 간 결제 동기 구조가 직관적  
- PaymentWebhookController 로 비동기 정산 처리 가능  
- Order-Service에서 paymentKey 관리/검증하기 쉬움

# 📌 결론

> **"한국 개발 환경에서 가장 빠르고 안정적으로 구현 가능한 테스트 PG = Toss Payments"**

따라서  
이번 Retail 모노레포 MSA 구조에서는 Toss Payments 사용이 최선의 선택이며,  
실제 운영 환경으로 확장할 때에도 API 변경 없이 그대로 확장 가능하다는 장점이 있음.
